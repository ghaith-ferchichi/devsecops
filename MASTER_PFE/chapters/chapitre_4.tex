\section{Implémentation de la solution}

    \subsection{Phase 1 — Fondations : VPS, Docker et webhook GitHub}

        La première phase du projet a consisté à mettre en place l'ensemble de
        l'infrastructure Docker \cite{Docker} et à valider la réception des webhooks GitHub. La
        configuration de performance d'Ollama est critique pour un fonctionnement CPU-only.
        Sans tuning, le modèle 14B consomme plus de onze Go de mémoire KV à seize mille
        tokens de contexte avec les paramètres par défaut. La figure \ref{fig:ollama_compose}
        présente la configuration Ollama dans le fichier \texttt{docker-compose.yml}.

        \begin{figure}[H]
            \centering
            \frame{\includegraphics[width=0.88\columnwidth]{img/ollama_docker_compose}}
            \caption{Configuration Ollama dans docker-compose.yml — tuning CPU Haswell}
            \label{fig:ollama_compose}
        \end{figure}

        La variable \texttt{OLLAMA\_FLASH\_ATTENTION=1} réduit la complexité mémoire du
        cache KV de O(n²) à O(n), économisant environ 650 Mo à seize mille tokens de
        contexte. La variable \texttt{OLLAMA\_KV\_CACHE\_TYPE=q8\_0} applique une
        quantisation huit bits au cache KV, divisant sa consommation mémoire par deux.
        La mémoire partagée est fixée à deux Go pour les buffers de synchronisation
        des threads, contre une valeur par défaut de soixante-quatre Mo insuffisante
        pour douze cœurs parallèles. La limite mémoire de quarante-deux Go protège les
        autres services du VPS tout en permettant au modèle 14B (neuf Go) de fonctionner
        avec une marge suffisante.

        Lors de l'intégration du webhook GitHub, un défi technique a été rencontré :
        nginx configuré en HTTP/1.0 par défaut manglait les corps chunked des requêtes
        GitHub, provoquant systématiquement des erreurs 500. La figure \ref{fig:nginx_fix}
        présente le correctif appliqué dans la configuration nginx.

        \begin{figure}[H]
            \centering
            \frame{\includegraphics[width=0.88\columnwidth]{img/nginx_webhook_fix}}
            \caption{Correctif nginx — proxy HTTP/1.1 pour les webhooks GitHub}
            \label{fig:nginx_fix}
        \end{figure}

        L'ajout de \texttt{proxy\_http\_version 1.1} et \texttt{proxy\_set\_header Connection ""}
        a résolu ce problème en forçant nginx à utiliser des connexions persistantes
        HTTP/1.1 avec l'agent, éliminant le mangling des corps chunked.

    \subsection{Phase 2 — Pipeline de sécurité : scanners et LLM}

        \subsubsection{Déploiement de node-exporter}

        Le déploiement de node-exporter \cite{NodeExporter} a nécessité une configuration

        \texttt{network\_mode: host} pour accéder aux systèmes de fichiers \texttt{/proc}
        et \texttt{/sys} du host. Une règle iptables a été ajoutée pour autoriser le
        bridge Docker à joindre le port 9100 du host. La figure \ref{fig:node_exporter}
        présente la configuration déployée.

\section{Tests de validation}

    Afin de vérifier le bon fonctionnement de chaque mécanisme de sécurité et de résilience
    de la solution, nous avons conduit une série de tests unitaires sur les composants
    critiques de la plateforme. Ces tests reproduisent des conditions réelles d'erreur ou
    d'attaque pour valider les comportements attendus.

    \subsection{Test de la validation HMAC-SHA256}

        Le premier test vérifie que l'agent rejette correctement les webhooks dont la
        signature est invalide ou absente. La figure \ref{fig:test_hmac} présente le
        résultat du test : un webhook avec une signature incorrecte reçoit un code HTTP
        403 Forbidden, tandis qu'un webhook correctement signé est accepté avec un code 202.

        \begin{figure}[H]
